1) บทบาทของวิศวกรซอฟต์แวร์ (Software Engineer Role)

วิศวกรซอฟต์แวร์มีหน้าที่ออกแบบและพัฒนาระบบซอฟต์แวร์ให้ตอบโจทย์ความต้องการของผู้ใช้และองค์กร ตั้งแต่การเก็บ Requirement การออกแบบโครงสร้างระบบ ไปจนถึงการเขียนโค้ดและทดสอบระบบ นอกจากนี้ยังต้องแก้ไขปัญหา บำรุงรักษาระบบ และทำงานร่วมกับทีมอื่นเพื่อให้การพัฒนาเป็นไปอย่างมีคุณภาพและตรงเวลา รวมถึงต้องอัปเดตความรู้และเทคโนโลยีใหม่ ๆ อย่างสม่ำเสมอ

### ตัวอย่างงานที่วิศวกรซอฟต์แวร์ทำ
- วิเคราะห์ Requirement
- ออกแบบระบบ (Architecture, Database, API)
- เขียนโค้ด และทดสอบระบบ
- Review code และแก้ไขบั๊ก
- ประสานงานกับทีมอื่น เช่น PO/QA/DevOps

### บทบาทอื่นในทีมพัฒนา
- **QA Engineer** – ทดสอบระบบ หา defect และยืนยันคุณภาพงาน
- **DevOps Engineer** – ทำ CI/CD, ดูแล server, automate deployment
- **Product Owner (PO)** – กำหนดความต้องการของฟีเจอร์ และจัดลำดับความสำคัญของงาน
- **Business Analyst (BA)** – เก็บ requirement จากลูกค้าและเขียนสเปคงาน
- **UI/UX Designer** – ออกแบบหน้าตาและประสบการณ์ผู้ใช้ของระบบ

2) ภาพรวม SDLC (Software Development Life Cycle)
Requirements → Design → Implementation → Testing → Deployment → Maintenance

### 1. Requirements  
เก็บ วิเคราะห์ และสรุปความต้องการของระบบให้ชัดเจน เพื่อเป็นรากฐานในการออกแบบและพัฒนา หาก requirement ไม่ครบหรือไม่ถูกต้องจะส่งผลต่อทั้งระบบ

### 2. Design  
ออกแบบสถาปัตยกรรมระบบ ฐานข้อมูล และการทำงานของฟีเจอร์ต่าง ๆ ให้มีความสอดคล้องกัน การออกแบบที่ดีช่วยลดบั๊กและเพิ่มความเร็วในการพัฒนา

### 3. Implementation  
ลงมือเขียนโค้ดตามแบบที่ออกแบบไว้ ใช้มาตรฐานการเขียนโค้ดที่ดีและจัดการเวอร์ชันด้วย Git อย่างเป็นระบบ

### 4. Testing  
ทดสอบระบบในหลายระดับ เช่น Unit Test, Integration Test, UAT เพื่อหาบั๊กและยืนยันว่าระบบทำงานถูกต้องก่อนส่งมอบ

### 5. Deployment  
นำระบบขึ้นใช้งานบน environment จริง (Production) ต้องจัดการความเสถียร ความปลอดภัย และเวอร์ชันของระบบให้เหมาะสม

### 6. Maintenance  
ดูแลระบบหลังใช้งานจริง เช่น แก้บั๊ก ปรับปรุงประสิทธิภาพ และเพิ่มฟีเจอร์ใหม่ ๆ เพื่อให้ระบบรองรับการใช้งานระยะยาว

3) Waterfall vs Agile – ตารางเปรียบเทียบ

| ประเด็น              | Waterfall                     | Agile (Scrum/Kanban)                     |
|----------------------|------------------------------|------------------------------------------|
| การวางแผน           | วางแผนครั้งเดียวแบบยาว      | วางแผนเป็นรอบสั้น ๆ (Sprint)           |
| การส่งมอบงาน        | ส่งเมื่อจบโครงการ           | ส่งเป็นส่วนย่อย (Increment) บ่อย ๆ     |
| การเปลี่ยนแปลง     | เปลี่ยนยาก และมีต้นทุนสูง   | ปรับได้ง่ายระหว่างการพัฒนา             |
| การสื่อสารกับลูกค้า | น้อยและเป็นทางการ           | สื่อสารบ่อย มี Sprint review            |
| ความยืดหยุ่น        | ต่ำ                          | สูง                                      |
| ความเหมาะสม        | เหมาะกับงานที่ requirement ชัดเจน | เหมาะกับงานที่ต้องปรับเปลี่ยนตลอด |

---

## การประยุกต์ใช้กับ Term Project ของกลุ่มเรา

### 1. ถ้า Term Project ใช้ Waterfall 100%

**จุดเด่น**
- แผนงานชัดเจน เพราะทำทีเดียวตั้งแต่ต้น  
- เหมาะกับงานที่ requirement ไม่เปลี่ยน  
- ตรวจสอบความคืบหน้าได้ง่ายตามลำดับขั้นตอน  

**ข้อจำกัด**
- ถ้า requirement เปลี่ยน ต้องกลับไปแก้หลายขั้นตอน  
- อาจเจอบั๊กหรือปัญหาใหญ่ตอนท้าย เพราะไม่มีการทดสอบระหว่างทาง  
- ได้งานช้า ไม่เห็นผลลัพธ์ย่อย ๆ ระหว่างการพัฒนา  

---

### 2. ถ้า Term Project ใช้ Agile แบบ Sprint

**จุดเด่น**
- เห็นงานจริงเป็นรอบ ๆ ทำให้ปรับปรุงได้เร็ว  
- รองรับการเปลี่ยนแปลง requirement ได้ดี  
- คุณภาพดีขึ้นเพราะมี feedback loop ทุก Sprint  

**ข้อจำกัด**
- ต้องประชุมและติดตามงานบ่อย  
- ต้องมีวินัยในการส่งงานทุก Sprint  
- ถ้าประเมินงานผิดอาจทำให้ Sprint ล้ม  

---

### 3. กลุ่มเราอยากเลือกแนวทางอะไร และเพราะอะไร?

กลุ่มเราต้องการใช้ **Hybrid Model (Waterfall + Agile)**  
คือ:

- ช่วงแรกใช้ Waterfall เพื่อเก็บ Requirement และออกแบบระบบให้ชัดเจน  
- หลังจากนั้นแบ่งการพัฒนาเป็น Sprint แบบ Agile เพื่อให้ปรับปรุงได้รวดเร็วและรับ Feedback ได้ทันที  

เหตุผลคือโปรเจกต์ของเราอาจมีการเปลี่ยนแปลงระหว่างพัฒนา และ Agile ช่วยให้ปรับได้ง่าย ขณะเดียวกัน Waterfall จะช่วยให้โครงสร้างและภาพรวมของระบบชัดเจนก่อนลงมือทำจริง
